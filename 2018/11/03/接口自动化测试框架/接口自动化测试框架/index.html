接口自动化测试框架

发表于 2018-11-03 | 分类于 Python

本文分享一个我写的接口自动化测试框架，这是一个比较简单且实用的接口自动化测试框架，支持socket、soap和http等多种接口协议。

接口自动化测试介绍

接口测试是测试系统组件间接口的一种测试。接口测试主要用于检测外部系统与系统之间以及内部各个子系统之间的交互点，测试的重点是要检查数据的交换，传递和控制管理过程，以及系统间的相互逻辑依赖关系等。

简而言之接口测试是一种黑盒测试，给待测程序一个符合规范的输入，然后校验返回的输出是否正确。随着程序版本迭代，新修改的接口或者配置可能会影响旧接口的使用，测试人员对旧接口仍需要进行回归测试。如果版本迭代较快或者旧接口较多，回归测试会浪费大量的人力物力和时间，使用接口自动化测试能很好地解决这一问题。

一个完整的接口自动化测试框架一般需要包含以下内容：

  • Log：测试日志，包括测试用例信息、待测程序启停情况、构造的测试数据、发送和接收的报文、校验信息、异常信息等。
  • Report：测试报告，包括测试用例总数、成功和失败的数量，测试时间。
  • Configure：配置文件，包括待测程序IP和端口、测试用例路径、接口协议配置等。
  • Common：公共函数库。
  • TestCase：测试用例。
  • TestData：测试数据。
  • Statistics：测试统计(可选项)，包括每次测试结果统计、对比、分析，为程序优化和流程改进提供参考。

现在市面上做接口自动化测试的工具很多，比较常见的有Jmeter+Ant、Java+Junit、Python+unittest，我原本打算使用Python+unittest这套自动化测试框架来进行接口测试，但这套框架的每个测试用例都需要用Python来写，不符合我们部门的期望。我自己写了个自动化测试框架，把测试用例和自动化测试框架区分开，把每个测试用例写进一个文本文件（可自定义文件后缀），然后用Python来进行分析和处理。

这样做有两个好处：
一、写测试用例的人不需要会Python，只需要会按规则写测试用例，起点低。
二、测试用例与语言无关，对自动化框架的依赖性比较小，若Python写的自动化框架有问题或者瓶颈，完全可以用Java或者Ruby重写自动化框架来执行这些测试用例。

框架介绍

框架流程图

我们的接口自动化测试框架流程如下图所示，程序运后，main方法依次读取每个测试用例，对每个测试用例调用analyCase方法来进行分析和处理，处理流程大致为：启动待测程序，与待测程序建立连接，构造和发送预置数据，发送和接收测试报文，校验测试报文，清理测试数据；处理完成后根据校验情况来记录测试结果；当所有测试用例执行完，统计测试结果。

analyCase方法

框架的思路是约定测试用例和analyCase方法之间不同的关键词，测试用例按照对应操作的关键词来写，analyCase方法按照对应操作的关键词来分析。例如：下图中的Case、Change、Do、Check为约定的部分关键词。

analyCase方法会逐行读取测试用例进行处理，当匹配到关键词时调用对应的处理函数，类似于事件监听器。我们将各种处理函数写在analyCase方法外，这样框架逻辑会很清晰，便于修改和维护，同时增强了可扩展性。

下图为一个简单的样例，真实的自动化框架会包含更多的关键词，不同的接口或者不同的校验方法需要用到不同的关键词，继而需要对不同的关键词进行对应的处理和对应的处理函数。

Case

对照analyCase方法来看一个简单的测试用例，如下图所示：

  • Case和Desc关键词是测试用例标题和描述，一般打印出来即可。
  • Change关键词用于修改配置文件，同时可以进行测试准备工作，例如启动待测试程序，备份，清除会干扰的测试数据等。
  • Do关键词用于构造测试数据，建立连接，发送预置数据，发送和接收报文等。
  • Check关键词用于校验返回的结果是否符合预期。
  • ENDCASE表示该用例已结束。

每个测试用例执行完都会进行统计，恢复配置文件，清除测试数据。

日志和报告

自动化测试需要在日志中记录很多关键信息，以便于测试用例不通过时查看具体原因，如下图所示：日志中需记录正在执行的测试用例名称，修改的配置文件，构造的测试数据，发送的测试报文，接收的返回报文，测试结果等。

总结

这套接口自动化测试框架支持多种通信协议，现已应用到公司多个个项目的自动化测试当中。测试用例不依赖语言，不会Python也能写测试用例，可跨操作系统使用。框架内的处理函数和校验方法相互独立，不同的产品可以根据需要来进行增加和修改，同时也便于维护。

这套接口自动化框架比较依赖于测试产品，需要根据产品不同的连接方式，不同的接口进行定制关键词和处理函数。

另外，若测试用例比较多且测试用例之间互不影响，可以多线程运行不同测试用例，可大幅度提高测试速度。另外还可以追加测试报告美化，测试报告自动发送邮件功能。
